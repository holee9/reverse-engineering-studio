# 01. Methodology

## 핵심 원칙

### P1. 점진적 레벨링
각 레벨은 **독립 릴리즈 가능한 단위**. L2 완료 시점에 L3~L5 없이도 실사용 가능해야 함.

### P2. 이슈 드리븐
- 모든 코드 변경은 이슈 번호와 연결
- 5분 초과 작업은 이슈 생성
- 이슈 = 작업 기억 + 의사결정 로그

### P3. 최소 단위
- 1 Task = 최대 1일 (8시간 상한)
- 초과 예상 시 반드시 분할
- 분할 불가 → Epic으로 승격

### P4. 체크포인트 게이팅
레벨 완료 시 다음 질문 통과 전에는 다음 레벨 진입 금지:
- [ ] 현재 레벨 결과물로 실제 DLL 분석이 가능한가?
- [ ] 다음 레벨 투자가 정말 필요한가, 아니면 현재 레벨 개선이 더 가치 있는가?
- [ ] 현재 코드 품질이 확장을 감당하는가?
- [ ] CI 파이프라인이 안정적으로 동작하는가?

### P5. CI 통과 필수
- CI 실패 PR은 머지 금지
- main 브랜치는 항상 빌드 가능 상태
- 릴리즈 태그는 CI 성공 후에만 생성

## 개발 사이클

### 일일 사이클
```
1. Issue 선정 (In Progress 라벨)
2. 브랜치 생성: feature/{issue-num}-{slug}
3. 구현 + 커밋 (이슈 번호 참조)
4. 로컬 빌드·테스트 통과 확인
5. Push → PR 생성
6. CI 통과 확인
7. Self-review → Merge
8. 이슈 Close
```

### 주간 사이클
```
월 09:00 — 주간 계획 (이번 주 Close할 이슈 선정)
화~목   — 이슈 기반 구현
금 16:00 — 주간 회고 (완료 이슈 리뷰, 차주 백로그 정제)
```

### 레벨 사이클
```
[Planning]       Epic 정의, Task 이슈 생성 (1일)
[Setup]          환경 준비, CI 검증 (0.5일)
[Implementation] 태스크 순차 처리 (70%)
[Integration]    실제 DLL 테스트 (20%)
[Checkpoint]     DoD 검증 (10%)
[Release]        태그 생성 → CI 릴리즈 빌드
```

## 이슈 세분화

### 좋은 이슈의 조건
- **단일 책임**: 1개 이슈 = 1개 목적
- **측정 가능**: 완료 판단 객관적
- **크기 제한**: 2~8시간
- **DoD 명시**: 체크박스로 완료 기준 명문화

### 이슈 분할 휴리스틱
| 규모 | 처리 |
|------|------|
| < 30분 | 이슈 불필요, 커밋에만 기록 |
| 30분 ~ 8시간 | Task Issue |
| 8시간 ~ 1주 | 분할 후 Task Issues |
| > 1주 | Epic + 하위 Tasks |

### 나쁜 이슈 예시
- ❌ "L2 구현" — 너무 큼 → Epic으로
- ❌ "버그 수정" — 대상 불명확
- ❌ "리팩토링" — 범위 불명확

## 위험 관리

| 위험 | 대응 |
|------|------|
| Ghidra 통합 난항 | L4 전 Spike 이슈로 선행 검증 |
| 이슈 관리 부담 | 주간 정제, 30일 미진행 이슈 Close |
| 번아웃 | 레벨 단위 릴리즈로 성취감 확보 |
| Scope Creep | "Out of Scope" 엄격 준수 |
| CI 불안정 | L1부터 CI 세팅, 매 레벨 CI 리뷰 |
