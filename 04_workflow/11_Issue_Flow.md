# 11. Issue Lifecycle

## 이슈 상태 흐름

```
┌─────────┐   ┌────────────┐   ┌─────────────┐   ┌───────┐   ┌────────┐
│ Backlog │──►│ Ready      │──►│ In Progress │──►│ Review│──►│ Closed │
└─────────┘   └────────────┘   └─────────────┘   └───────┘   └────────┘
     ▲              │                 │              │           │
     │              │                 ▼              │           │
     │              │           ┌──────────┐         │           │
     │              │           │ Blocked  │         │           │
     │              │           └──────────┘         │           │
     │              │                                            │
     └──────────────┴────────────────────────────────────────────┘
                     (재오픈, 백로그 재진입)
```

## 상태 정의

### 1. Backlog
- 이슈 생성 직후 기본 상태
- 제목·설명 작성되었으나 DoD 미완성 가능
- 착수 여부 미결정

### 2. Ready
- DoD 완성
- 관련 이슈·의존성 명확
- 누구나 (=본인이) 즉시 착수 가능
- 다음 스프린트 후보

### 3. In Progress
- 현재 작업 중 (1인 개발이므로 최대 3개 권장)
- 브랜치 생성됨
- WIP 커밋 진행 중

### 4. Review
- PR 생성됨
- CI 실행 중 또는 통과
- Self-review 대기

### 5. Blocked
- 외부 의존성·결정 대기
- 블로커 이슈 존재
- **반드시 차단 원인 코멘트 남기기**

### 6. Closed
- PR 머지됨
- DoD 모두 통과
- 이슈 자동 Close (커밋 메시지 `Closes #N`)

## 이슈 라이프사이클 상세

### 생성 단계

#### 좋은 이슈 생성 체크리스트
- [ ] 제목은 40자 이내, 동사로 시작
- [ ] 본문에 **Why** (왜 필요한가) 명시
- [ ] **What** (무엇을 해야 하는가) 구체적
- [ ] **DoD** 체크박스로 작성
- [ ] 예상 시간 명시 (Estimate)
- [ ] 관련 라벨 부착
- [ ] 마일스톤 할당 (해당 시)
- [ ] Parent Epic 링크 (Task인 경우)

#### 제목 예시
- ✅ `Add PE file loader using PeNet`
- ✅ `Fix null reference when DLL has no exports`
- ❌ `PE 로더` — 너무 짧음, 동사 없음
- ❌ `PE 관련 이슈들 처리` — 범위 불명확

### 준비 단계 (Backlog → Ready)

**체크리스트**:
- [ ] DoD 모든 항목 구체적
- [ ] 기술적 접근 방법 메모 (코멘트로)
- [ ] 차단 요인 없음 확인
- [ ] 다음 2~3 이슈 중 하나면 상세화

### 진행 단계 (Ready → In Progress)

**착수 시점 할 일**:
1. 라벨 `status:in-progress` 추가
2. Assignee 설정 (본인)
3. 브랜치 생성: `feature/<num>-<slug>`
4. 첫 커밋에 이슈 번호 참조

### 리뷰 단계 (In Progress → Review)

**PR 생성 시**:
1. PR 설명에 `Closes #N` 포함
2. CI 통과 확인
3. DoD 체크리스트 PR에서도 검토
4. Self-review (변경된 파일 전체 다시 읽기)

### 종료 단계 (Review → Closed)

**Merge 시 자동 처리**:
- 이슈 자동 Close (`Closes #N` 덕분)
- 브랜치 자동 삭제 (Settings 설정)

**수동 처리**:
- 관련 문서 업데이트 여부 확인
- 후속 이슈 필요 시 생성

## 이슈 우선순위 결정

### 3-Dimension Matrix

```
          긴급함                      긴급하지 않음
        ┌──────────────────────┬───────────────────┐
중요    │  priority:high        │  priority:medium  │
        │  즉시 처리            │  계획적 처리       │
        ├──────────────────────┼───────────────────┤
덜중요  │  priority:medium      │  priority:low     │
        │  위임 고려 (해당없음)   │  나중, 혹은 폐기    │
        └──────────────────────┴───────────────────┘
```

### 우선순위별 처리 원칙

| Priority | SLA | 예시 |
|----------|-----|------|
| `high` | 1~3일 내 | 레벨 DoD 필수, CI 실패, 블로커 |
| `medium` | 1~2주 내 | 일반 기능, 사용성 개선 |
| `low` | 백로그 | Nice-to-have, 문서 다듬기 |

## WIP (Work In Progress) 제한

**1인 개발 권장**:
- `status:in-progress` 이슈: 최대 **3개**
- `status:review` 이슈: 최대 **2개**
- 이 한도 초과 시 신규 착수 금지, 기존 마무리 우선

---

## 이슈 정리 주기

### 매주 (금요일 회고 시)
- 1주 이상 `status:in-progress`이나 커밋 없는 이슈 → 원인 확인
- `Blocked` 상태 이슈의 차단 해제 가능성 검토
- 다음 주 착수할 이슈 3~5개를 `Ready`로

### 레벨 종료 시
- 해당 마일스톤 미완료 이슈 → 다음 마일스톤 재할당 또는 `wontfix`
- 누락된 후속 이슈 추가

### 매월
- 30일 이상 `Backlog` 상태 이슈 검토 → 여전히 유효한지 확인
- 중복 이슈 병합
- 실효성 없는 이슈 `wontfix` 라벨 후 Close

## 이슈 참조 규약

### 커밋 메시지
- `Closes #123` — 머지 시 자동 Close
- `Fixes #87` — 버그 이슈
- `Refs #45` — 참조만

### 이슈 간 연결
- 본문에 `Part of #E4` — Epic 하위 Task
- 본문에 `Depends on #50` — 선행 이슈
- 본문에 `Blocks #75` — 후행 이슈

### Cross-reference 자동화
- `#N` 언급 시 양쪽 이슈에 자동 링크
- Timeline에 기록되어 추적 용이

## 1인 개발 팁

- **컨텍스트 복원 우선**: 이슈 코멘트에 생각·막힌 지점 자주 남기기 (미래의 내가 감사할 것)
- **WIP 커밋 허용**: 로컬에서는 자유롭게, PR 전 정리
- **주간 회고 필수**: 월 4회 × 15분 = 1시간 투자로 방향성 유지
- **Epic 쪼개기**: Epic에 10개 이상 Task 쌓이면 쪼개기 검토
